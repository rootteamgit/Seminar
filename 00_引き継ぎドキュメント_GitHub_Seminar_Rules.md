## 🎯 このプロジェクトの目的

RootTeam のセミナー制作に関する **すべてのルール・制約・テンプレート** を GitHub で一元管理し、以下を実現：

1. **同じ指示を二度させない**: ルール更新 → 自動的に全スタッフが参照
2. **誰が指示を出してもルール化**: 新しい指示 → 即座に対応ファイル更新 → GitHub push
3. **スケーラビリティ**: このプロジェクトを使うすべての人が最新ルールを自動参照

---

## 📊 プロジェクト概要

### リポジトリ情報
- **リポジトリ名**: `rootteamgit/Seminar`
- **可視性**: **Private（設定予定）**
- **ブランチ**: `main` のみ
- **言語**: Markdown（ルール）+ 運用ドキュメント

### ファイル構成（最終形）

```
rootteamgit/Seminar/
├── README.md                      ← リポジトリ概要

└── rules/
    ├── MASTER_RULES.md            ← 全ルール集約（必ず参照）
    ├── kikakusho.md               ← 企画書作成ルール
    ├── slide_design.md            ← スライド設計ルール
    └── script.md                  ← 台本作成ルール
```

## 📝 ファイル詳細

### 1. MASTER_RULES.md（最重要）
**目的**: セミナー制作に関するすべての基本ルール・制約を集約

**主な内容**:
- 🚨 クリティカルルール
  - **TalkSlide 制約**: 動画の尺を指定してはいけない（自動判定）
  - **スライド孤立禁止**: すべてのスライドに台本上の位置が必須
  - **アポロジー原則**: パートナーへの誤りは謝罪で対応（ポジティブ転換NG）
  
- 📁 ファイル形式・出力先
  - 台本: `.docx` (Node.js `docx` ライブラリ)
  - 企画書: `.xlsx` (Python `openpyxl`)
  - スライド: `.pptx` (pptxgenjs)
  - CSV: `utf-8-sig` BOM付き
  - 出力先: `/mnt/user-data/outputs/`

- 🎨 スライドデザインシステム
  - 10×5.625 inch / 16:9
  - 背景: #0D1B2E (ネイビー)
  - アクセント: #29C5F6 (シアン), #EC2D7B (ピンク)
  - フォント: Yu Gothic
  - マージン: 0.55 inch 定数

- 📝 台本記法（統一記号）
  - 🟩 = 森永（RootTeam COO）
  - 🟦 = 上出さん／外部登壇者
  - 📊 = スライド切替（番号必須）
  - 📝 = 運営メモ
  - 🔴 = 要確認項目
  - ⏱️ = タイミング
  - 🎬 = ライブデモ

- 📊 セミナー設計知見
  - 高パフォーマンステーマ: HR×AI×コスト削減
  - 危険ゾーン: 開始30秒、7～9分地点
  - 動画セグメント: 6分以内
  - エコーチェンバー防止: 毎回テーマの多様性チェック

- 固定出演者・パートナー情報
- デモ素材品質基準

### 2. kikakusho.md
**目的**: セミナー企画書（Excel）作成のすべてのルール

**主な内容**:
- 企画書の 7 つの必須フィールド
  1. セミナータイトル
  2. リード文（「〇〇で悩んでいませんか？」から始まる）
  3. ターゲット（職種・役職・悩み）
  4. アジェンダ（時間配分）
  5. 解決できる課題（具体的成果）
  6. 登壇者・会社情報
  7. 集客目標・CTA・アーカイブ有無

- テーマ確定の 6 ステップフロー
  - STEP 1: 登壇者プロファイルを固める
  - STEP 2: ターゲットを設定
  - STEP 3: ブランド（約束）を言語化
  - STEP 4: テーマ候補を 3 分類で出す
  - STEP 5: ブランドとの掛け算でテーマを絞る
  - STEP 6: 過去データと照合して確定

- 高パフォーマンステーマの条件（実績ベース）
- リード文の書き方・テンプレート
- エコーチェンバー防止チェックリスト
- 共催パートナーがいる場合の記述方法
- 🔴 未確定フラグの使い方

### 3. slide_design.md
**目的**: PowerPoint スライド（.pptx）設計・デザインのすべてのルール

**主な内容**:
- デザインシステム仕様

  - カラーパレット: ネイビー / シアン / ピンク
  - フォント: Yu Gothic
  - マージン: 0.55 inch 定数

- スライドテンプレート 3 パターン
  - タイトルスライド
  - コンテンツスライド
  - 画像・デモスクリーンショット

- スライド設計の基本原則
  - 1 分 1 スライドの「適度な密度」
  - テキストサイズの階層
  - 色使いのルール
  - 図表・チャートの仕様

- NG パターンと修正案
- デモスクリーンショット・画像の品質基準
- ライブデモ用スライドの作り方
- pptxgenjs での自動生成時の注意

### 4. script.md
**目的**: セミナー進行台本（.docx）作成のすべてのルール・記法

**主な内容**:
- 台本の目的（3 つ）
  - 本番中の指針
  - 進行管理
  - アーカイブ字幕用

- 台本記法（統一記号）
  - 🟩 / 🟦 / 📊 / 📝 / 🔴 / ⏱️ / 🎬

  - セクション 1: オープニング（0～5 分）
  - セクション 2: 背景・課題提示（5～20 分）
  - セクション 3: メイン解説 + デモ（20～50 分）
  - セクション 4: Q&A・まとめ（50～60 分）

- 台本の品質チェックリスト
  - スライド整合性
  - 登壇者確認
  - デモマーク
  - タイミング照合
  - 🔴 要確認ゼロ化

- スクリプト長さの目安（文字数と時間の相関）
- 話し言葉の作り方
- デモ・ライブ実演の記述方法
- ファイル命名規則
- 修正版（v2, v3）の管理方法

## 🔄 GitHub アップロード手順

### オプション A: GitHub web UI（手動・最速）

1. https://github.com/rootteamgit/Seminar を開く
2. **Add file** → **Create new file**
3. 以下のファイルを順番にコピペして作成：
   - `README.md`
   - `CHANGELOG.md`
   - `rules/MASTER_RULES.md`
   - `rules/kikakusho.md`
   - `rules/slide_design.md`
   - `rules/script.md`
4. 各ファイル作成時に **Commit directly to main branch**
5. 完了後、**Settings** → **Danger Zone** → **Change visibility** → **Private**

### オプション B: Grantry MCP（自動）

Grantry の GitHub 連携が有効な場合：
```
Grantry MCP → commit & push
ファイル: 6 つ
ブランチ: main
メッセージ: "Initial seminar rules setup"
```

## 🎯 アップロード後の運用フロー

### セミナー作成前の必須参照

**毎回の作業開始時**:
```
1. MASTER_RULES.md をfetch（常に最新版を確認）
   https://raw.githubusercontent.com/rootteamgit/Seminar/main/rules/MASTER_RULES.md

2. 作業タイプに応じて該当ファイルを参照
   - 企画書作成 → rules/kikakusho.md
   - スライド作成 → rules/slide_design.md
   - 台本作成 → rules/script.md
```

### 新しい指示が出たときの対応

**誰から・どのような指示でも同じプロセス**:

1. 指示内容を理解
2. 該当ファイルに反映
3. CHANGELOG.md に変更を記録
   ```
   ## YYYY-MM-DD | [ファイル名] | 指示者

   ### 追加/変更/削除
   - 内容
   ```
4. GitHub にpush（PAT または Grantry MCP）
5. 以降のセッションで自動参照

**例**:
- 新しいスライドデザイン規則が出た → slide_design.md 更新 → push
- 新しい台本記法が決まった → script.md 更新 → push
- 企画書フィールドが増えた → kikakusho.md 更新 → push

## ⚠️ 注意事項

1. **このリポジトリは Private にすること**
   - 社内ナレッジが含まれるため
   - GitHub Settings → Change visibility → Private

2. **PAT / Grantry 認証情報は管理**
   - リポジトリに絶対にコミットしない
   - 必要に応じてrevokeして新規作成

3. **ファイル命名規則を守る**
   - バージョン管理: `_v1.docx` → `_v2.docx`
   - 修正内容を記録に残す

4. **🔴 要確認フラグはゼロに**
   - 企画書・台本完成時に未確定項目ゼロであること

5. **GitHub fetch は毎回**
   - キャッシュ参照ではなく、常に raw.githubusercontent.com から最新版を取得
